BuildForge Agents

Локальная мультиагентная система на Python, которая берёт исходники проекта (Electron, C++, C#, Python, HTML), готовит окружение, собирает артефакты (в первую очередь Windows EXE / portable / installer), при ошибках пытается исправить их по правилам и памяти, и только потом считает задачу успешной.

Что это такое (по факту)

Это не «магический ИИ, который пишет любой софт». Это оркестратор сборки с набором агентов и builders. Точка входа — app.py. Центр — класс Orchestrator в core/orchestrator.py.

  • Вход: путь к папке проекта на диске.
  • Выход: список артефактов (пути к файлам) + лог + статус ok/fail.
  • Успех: только если после build есть реальные файлы-артефакты (truth chain). Пустой dist = fail.
  • Эволюция: развитие идей setuper / BuildDoctor в сторону агентной фабрики.

Что система умеет собирать

Тип (ProjectKind)Как детектитсяBuilderЧто делает
electron package.json + electron / electron-builder в deps или scripts agents/builders/electron_builder.py npm install, при необходимости ставит electron-builder, затем npm run dist|build или npx electron-builder --win. Ищет .exe/.msi в dist/release/out

csharp *.csproj / *.sln csharp_builder.py dotnet restore + dotnet publish win-x64 self-contained single file
cpp CMakeLists.txt / *.vcxproj cpp_builder.py cmake -S/-B (MSVC или MinGW) → cmake --build Release, копирует .exe в dist
html index.html / *.html html_builder.py Копирует статику в dist/html_app, делает run.bat (Edge/Chrome app mode), zip portable
Ограничение правды: HTML-сборка — это portable-обёртка (zip + bat), а не компиляция в нативный GUI-движок. C++/C#/Electron/Python завязаны на установленный toolchain на машине пользователя.

Зачем так устроено

Принципы, которые реально заложены в код и в neural_core/ — не маркетинг, а правила принятия решений.

1. Decomposition

Большая задача «собери проект» режется на стадии: detect → diagnose → prepare → build → fix → package → verify → remember. Каждая стадия — отдельный агент или метод builder.

2. Truth verification

Отчёт «успех» без файла на диске запрещён. В оркестраторе: if not ok or not artifacts → FAILED с текстом про truth chain. Verifier дополнительно smoke-запускает .exe.

3. Self-healing по слоям

Сначала memory (SQLite lessons), потом rules из knowledge/known_errors.json, потом (только в neural) LLM-команда. Не «сразу GPT».

4. Autopilot без обязательного LLM

Режим autopilot работает на правилах и builders. LLM — опция (neural), не зависимость.

5. Память между запусками

core/memory.py: lessons, facts, actions, messages. Повторяющиеся ошибки накапливают hits и предлагают тот же fix.

6. Защита ресурсов

ResourceGuard смотрит RAM через psutil и делает breath/wait, чтобы не ронять машину на тяжёлых build.

Архитектура системы

Структура репозитория (то, что лежит в GitHub сейчас):

buildforge-agents/ ├── app.py # CLI/GUI/phone entry (Typer) ├── core/ │ ├── orchestrator.py # главный цикл job │ ├── types.py # ProjectKind, Mode, Stage, BuildResult… │ ├── memory.py # SQLite lessons/facts/actions/messages │ ├── event_bus.py # pub/sub логов и status │ └── resource_guard.py # RAM guard ├── agents/ │ ├── base.py # run_cmd, log, which │ ├── detector.py │ ├── diagnostician.py │ ├── fixer.py │ ├── packager.py │ ├── verifier.py │ └── builders/ # python, electron, csharp, cpp, html ├── knowledge/ │ └── known_errors.json # regex → fix action ├── llm/ # ollama, openai-compat, gguf, router ├── neural_core/ # NAP++ / INTENTION_ENGINE тексты + system_prompt() ├── server/phone_api.py # FastAPI + WebSocket :8787 ├── ui/main_window.py # PySide6 ├── templates/ # NSIS / Inno jinja stubs ├── BuildForge.spec # PyInstaller spec самого BuildForge └── docs/index.html # эта страница

Поток данных

  1. Пользователь запускает app.py (GUI / CLI -p / --serve-phone).
  2. Создаётся Orchestrator: Memory, ResourceGuard, агенты, optional LLM.
  3. run(project_dir, mode, target) ведёт job и шлёт события в BUS (GUI и phone их слушают).
  4. Результат — BuildResult: ok, stage, artifacts[], log_tail, duration, kind.

Как работает пайплайн (код оркестратора)

DETECT→ DIAGNOSE→ PREPARE→ BUILD→ FIX-LOOP→ PACKAGE→ VERIFY→ REMEMBER

1. DETECT — DetectorAgent

Смотрит файлы в корне проекта. Приоритет примерно такой: Electron (package.json+electron) → C# → CMake/C++ → Python → HTML. Возвращает DetectResult(kind, entrypoint, confidence, hints). Если unknown — job сразу FAILED.

2. DIAGNOSE — DiagnosticianAgent

Проверяет toolchain в PATH: git, python, node/npm, dotnet, cmake, компилятор C++, makensis/iscc, pyinstaller. Blockers (обязательное отсутствует) останавливают job. Warnings — нет.

3. PREPARE — builder.prepare()

Зависит от стека: venv+pip, npm install, dotnet restore, cmake configure… При fail — один проход fix_loop и retry prepare.

4. BUILD — builder.build()

Собственно сборка. Должна вернуть (ok, log, artifacts[]).

5. FIX-LOOP — до max_fix_retries (по умолчанию 3)

FixerAgent.match(log, kind):

  1. Ищет урок в SQLite memory по signature ошибки.
  2. Иначе матчит regex из known_errors.json (module not found → pip_install, npm missing, defender Access denied, нет dotnet/cmake, TS/CS compile → llm_or_manual…).
  3. apply() выполняет действие: pip/npm install, powershell Defender exclusion, report_toolchain и т.д.
  4. В режиме neural, если rules не помогли — LLM просят одну Windows-команду вида CMD: ... и выполняют её через run_cmd.

6. PACKAGE — PackagerAgent

По target (exe / zip / exe+installer): zip артефактов; если есть .exe и есть makensis — пытается собрать NSIS installer. Нет NSIS — честно пишет, что будет только ZIP.

7. VERIFY — VerifierAgent

Для .exe: короткий Popen + terminate (smoke). Не .exe — skip. Это не полный e2e-тест UI.

8. REMEMBER

Пишет lesson pipeline_ok / build_failed, action в memory, финальный status DONE/FAILED.

Почему fix не бесконечный: счётчик retries + stop event. Нет «вечного цикла». LLM не вызывается в autopilot, если вы сами его не включили.

Ядро NEURAL_ARCHITECT_PREMIUM++

Каталог neural_core/ — текстовое и программное ядро личности/протоколов. В neural-режиме оно собирается в system prompt через system_prompt() в neural_core/__init__.py.

ФайлРоль
v8.txtПолный системный промт NAP++ ULTIMATE v8 (из neural-architect-premium-pages)
FULL_PROMPT.md + 00_…06_*.mdINTENTION_ENGINE v8.4.00: activation, identity, architecture, decomposition, truth chain, vector correction, commands
BUILDFORGE_PERSONA.mdПривязка ядра к задаче BuildForge: миссия сборки, обращение «Неиро», автор
__init__.pyKERNEL_NAME, KERNEL_VERSION, AUTHOR, CONTACTS, сборка prompt (persona + выдержки протоколов + excerpt v8)

Как протоколы ядра стыкуются с кодом

  • DECOMPOSITION → стадии Stage и порядок вызовов в Orchestrator.run.
  • TRUTH VERIFICATION CHAIN → обязательные artifacts + verifier; fail без файлов.
  • VECTOR CORRECTION → memory.lessons: ошибка → действие → success/hits, повторное использование.
  • SELF-HEALING → Fixer + known_errors + optional LLM CMD.

Источники ядра: v8 ULTIMATE · INTENTION_ENGINE v8.4.00

Режимы работы

ModeПоведениеLLM
autopilot Полный пайплайн на rules + builders + memory. Основной режим CLI по умолчанию. Не нужен
manual Тот же оркестратор, управление через GUI/CLI (кнопки, явный запуск). Опционально
neural При blockers/fail подключает chat LLM; fix может выполнить CMD: из ответа. GUI chat → neural_chat(). Ollama / OpenAI-compatible / GGUF

LLM backends (llm/router.py)

  • ollama — HTTP /api/chat (по умолчанию http://127.0.0.1:11434, модель из env/флага).
  • openai / lmstudio — OpenAI-compatible chat completions.
  • gguf — локально через llama-cpp-python (если установлен) + путь к модели.
  • none — LLM выключен.

Конфиг-пример: .env.example (BUILDFORGE_MODE, BUILDFORGE_LLM, BUILDFORGE_MODEL, BUILDFORGE_MAX_FIX_RETRIES, phone port).

Интерфейсы

  • CLI — Typer в app.py: -p, -m, --llm, -t, --gui, --serve-phone.
  • GUI — ui/main_window.py (PySide6): выбор папки, mode/llm/target, лог BUS, chat, кнопка phone.
  • Phone API — server/phone_api.py: FastAPI + WebSocket, страница управления, POST /api/build, /api/stop, status/logs. Слушает 0.0.0.0:8787 (LAN).

Зависимости и сборка самого BuildForge

requirements.txt: PySide6, fastapi, uvicorn, websockets, httpx, pydantic, jinja2, psutil, rich, typer.

Сборка своего EXE: build_exe.bat + BuildForge.spec (PyInstaller, подхватывает neural_core/knowledge/templates).

Честно: для сборки чужих проектов на машине всё равно нужны их toolchain’ы (Node, .NET SDK, CMake/compiler, NSIS…). BuildForge их не заменяет — он их вызывает и чинит типичные сбои установки зависимостей.
